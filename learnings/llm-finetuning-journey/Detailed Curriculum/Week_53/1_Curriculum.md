## Concepts

### The LIMA Principle: Less Is More for Alignment

The LIMA paper (Zhou et al., 2023) trained a 65B LLaMA model on exactly 1,000 hand-selected examples and found it matched or exceeded models trained on orders-of-magnitude more data on many benchmarks. The mechanism: most of what a model needs to learn — style, format, safety, helpfulness — is already latent in the pretrained weights. Alignment fine-tuning just surfaces it.

The LIMA insight is not that small datasets always win. It is that **diversity and quality dominate raw count**. Their 1,000 examples covered 19 distinct tasks, had near-zero redundancy, and were individually verified by humans. A 10,000-example dataset where 8,000 examples are paraphrases of 50 templates will underperform LIMA.

For your domain, the analogous principle: 5,000 SQL examples covering all PostgreSQL/TimescaleDB features — window functions, CTEs, lateral joins, time_bucket, continuous aggregates, hyperfunctions, partial aggregates — will likely outperform 30,000 basic SELECT/WHERE/JOIN pairs.

**Operationalizing quality for SQL:**

A high-quality SQL example must satisfy all of:
1. The natural language question is unambiguous and domain-realistic
2. The SQL answer is correct — it executes without errors against the target schema
3. The SQL uses the most idiomatic approach (not just any query that happens to return the right rows)
4. The schema context provided is sufficient to answer the question (no missing table definitions)
5. The example represents a skill that is not over-represented elsewhere in the dataset

### Tulu 3: The Modern Post-Training Recipe

The Tulu 3 paper (Lambert et al., 2024) describes the post-training pipeline for AllenAI's open instruction-following models. Its data methodology is the most rigorously documented open recipe available as of 2025. Key lessons:

**Skill coverage taxonomy.** Tulu 3 defines a discrete set of skills (math, coding, reasoning, safety, instruction-following, etc.) and tracks coverage explicitly. During data selection, they ensure no skill is crowded out. For your dataset, your "skills" are SQL constructs: aggregation, window functions, time-series, joins, subqueries, CTEs, EXPLAIN/ANALYZE, schema introspection, conditional expressions, array operations, JSON/JSONB, full-text search, and TimescaleDB-specific constructs.

**Deduplication at multiple granularities.** Tulu 3 deduplicates at the prompt level (exact + near-duplicate removal), at the skill level (balanced), and removes contamination against known benchmarks (Spider, BIRD). You must do the same — if any of your training examples overlap with Spider or BIRD test sets, your evaluation numbers will be meaningless.

**Quality signals used by Tulu 3:**
- Perplexity under a base model (high perplexity = unusual/novel, low perplexity = mundane or memorized)
- Deita-style scoring (complexity + quality)
- IFEval-style instruction-following verification

**The on-policy vs. off-policy distinction.** Tulu 3 noted that data generated by stronger models (teacher models) is off-policy for your 7B student. If the teacher outputs stylistically differ too much from your student's natural generation style, training can be unstable. This is why Week 55's filtering step matters — you remove not just incorrect examples but also examples the model cannot meaningfully learn from.

### Designing Your Quality Strategy for Dataset v3

You enter this week with a v2 dataset from Phases 4–5. Your job now is to define precisely what v3 looks like before generating a single token.

**Step 1: Skill taxonomy audit.** List every SQL/PostgreSQL/TimescaleDB construct you care about. Estimate how many examples of each type you have in v2. Identify gaps.

**Step 2: Schema diversity audit.** How many distinct schemas (table structures) appear in your current data? Low schema diversity causes the model to memorize column names rather than learn SQL reasoning. Target at least 50 distinct schemas, ideally 100+.

**Step 3: Difficulty stratification.** Classify examples into Easy (single table, simple filter), Medium (multi-table join, aggregation), Hard (CTEs, window functions, nested subqueries), Expert (TimescaleDB hyperfunctions, continuous aggregate maintenance, EXPLAIN-driven optimization). Aim for roughly 20/35/30/15 split.

**Step 4: Source diversity.** v3 should draw from:
- Spider/BIRD augmented examples (rewritten for PostgreSQL dialect)
- Your hand-curated domain examples
- CoSQL/SParC multi-turn examples (Week 56)
- Synthetic examples from teacher model (Week 54)
- Stack Overflow Q&A pairs (filtered)

**Step 5: Write a one-page data card before generating.** Data cards force you to articulate assumptions. Include: intended use, dataset size, skill distribution, schema sources, quality criteria, known limitations.

### Common Misconceptions and Pitfalls

**"More data is always better."** False at the fine-tuning stage. You already have a pretrained model that understands language and SQL. You are calibrating its output style and domain knowledge. Garbage in = garbage out, and the garbage doesn't average away.

**"Execution correctness is the only quality signal I need."** Necessary but not sufficient. A query can execute correctly on your test database but be semantically wrong (returns the right shape but wrong rows for a subtly different schema). Add human review for at least a random 5% sample.

**"Deduplication is optional."** Near-duplicates cause the model to overfit to specific phrasings. A dataset with 1,000 unique examples and 49,000 paraphrases is effectively a 1,000-example dataset with inflated epoch count.

**"I don't need to track benchmark contamination."** If your training data contains Spider test set questions (even paraphrased), your Spider evaluation numbers are inflated and meaningless. Run MinHash similarity between your training set and Spider/BIRD test sets before finalizing.

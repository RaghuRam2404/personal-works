## Q1. Answer: C

**Answer:** C — Verify that the reference model weights are truly frozen.

**Why:** If both `rewards/chosen` and `rewards/rejected` are increasing together, the log-ratio (log π_θ − log π_ref) is increasing for all completions, not selectively for chosen ones. This happens when the reference model is being updated — if π_ref is changing along with π_θ, the log-ratio stays near zero for both, and then any increase in π_θ increases both ratios equally. Verify with `for p in ref_model.parameters(): assert not p.requires_grad`.

**Why others are wrong:**
- A: Increasing β would keep the ratio near 1, not cause both to increase.
- B: Lower LR would slow all changes, not fix the directional problem.
- D: Unsloth and vanilla TRL compute the same loss; switching implementations does not fix a frozen-weights bug.

## Q2. Answer: B

**Answer:** B — The preference dataset contains mostly simple query pairs; DPO has no signal for complex patterns.

**Why:** DPO is an offline method. It can only improve on patterns present in the training preference data. If your Week 44 dataset was generated by comparing two models on simple prompts (simple prompts are easier to generate and have higher labeling success rates), then the learned signal is concentrated on simple patterns. Complex queries generate mostly both-fail pairs during labeling (high discard rate), so few complex preference pairs make it into the dataset. This is exactly the distributional limitation of DPO, and it motivates GRPO's online approach.

## Q3. Answer: B

**Answer:** B — Some preference pairs are mislabeled — rejected SQL has higher log-probability under the training model than chosen SQL.

**Why:** DPO loss = −log σ(β·reward_margin). When reward_margin is negative (the model assigns higher probability to rejected than chosen), the loss is −log σ(negative) = −log(small number) which is a large positive number — the loss goes UP. But "loss goes negative" means σ(reward_margin) > 1, which is impossible for a valid sigmoid. What actually happens is the loss approaches zero from above, but may appear to bounce around zero due to data batch variation. Negative loss values in practice usually indicate a numerical issue or mislabeled pairs where chosen is truly less probable.

## Q4. Higher Execution but Lower Semantic Accuracy

Execution accuracy measures whether the SQL runs without error (syntactically and runtime-valid). Semantic accuracy measures whether it returns the correct rows. A model can have high execution accuracy and low semantic accuracy if it generates SQL that executes but with wrong logic — for example, `SELECT * FROM orders` always executes but is semantically wrong for "show orders over $100."

This implies your preference dataset labeled execution success (runs without error) as "chosen" without verifying semantic correctness (returns the right rows). The DPO model learned to prefer SQL that executes, not SQL that is correct. This is the "30% of chosen examples execute but return wrong rows" problem from Week 44's quiz scenario. Fix: rebuild the preference dataset with semantic verification using reference SQL.

## Q5. Refusal Rate Increased After DPO

**Diagnosis:** Your "rejected" SQL completions in the preference dataset likely included patterns that semantically resemble refusals — for example, rejected examples that contain no actual SQL (just error messages or explanations like "Error: table not found"). DPO's loss reduces the probability of all tokens in rejected completions. If error messages and non-SQL text appear frequently in rejected examples, the model learns to suppress any text that starts with a non-SQL pattern, which inadvertently suppresses legitimate refusals too — or conversely, if "I cannot..." appeared as a rejected completion, the model is being penalized for generating any response that starts that way.

**Concrete fix:** Filter the preference dataset to ensure all rejected examples are actual SQL (even if broken), not text explanations. If a candidate model said "I cannot generate SQL for this" and you included that as "rejected", remove that pair from the dataset. Run DPO again with the cleaned dataset.

## Q6. Using DPO Eval Results to Design GRPO Reward

DPO v2 failure modes and their GRPO reward implications:

**Failure mode 1: Complex queries not improved.** GRPO reward design: assign reward = 1 specifically for queries that execute correctly AND return the correct row count. By generating K=8 candidates for hard prompts and rewarding the best one, GRPO can explore novel SQL structures at training time that were not in the DPO dataset. Use v2 as the GRPO starting point so GRPO only needs to fix what DPO could not.

**Failure mode 2: Execution-correct but semantically wrong.** GRPO reward design: use reference SQL row comparison as the reward signal (not just execute/fail). A query that executes but returns 15 rows when the correct answer is 12 rows gets partial credit (0.5) rather than full credit (1.0). This pushes the model beyond mere execution-correctness.

**Failure mode 3: Empty results.** GRPO reward design: explicitly penalize empty results (reward = 0 if row_count = 0, even if the SQL technically executes). This prevents the model from learning that `SELECT NULL` is always "safe."

## Q7. DPO Failure — Systematic Diagnosis

**1. Low reward_margin plateau at 0.12:** The model cannot distinguish chosen from rejected in the dataset. The most common cause is that chosen and rejected SQL are too similar in log-probability under v1 (the reference model). If both SQL queries are nearly equally probable under v1, there is little gradient signal. This could be because: (a) the preference pairs are too easy (both are syntactically correct), (b) the pairs are mislabeled, or (c) β is too high and the regularization prevents the margin from growing.

**2. Oscillating loss (0.60–0.65):** The optimizer is finding no stable direction. The gradient from DPO is pointing in inconsistent directions across batches — indicating noisy preference labels (some chosen are actually worse than rejected in a way the model can detect), or the learning rate is too high for this regime (5e-7 is a starting point; try 1e-7).

**3. Three interventions ranked:**

Intervention 1 (highest impact): Data quality audit. Sample 50 random pairs and verify manually that chosen always executes correctly and rejected does not. If more than 15% are mislabeled, rebuild the preference dataset with stricter filtering. Low-quality data is the most common cause of DPO failure.

Intervention 2: Reduce β to 0.05. A reward_margin of 0.12 suggests the KL constraint is preventing the model from moving. Lowering β gives the optimizer more freedom to increase the margin. With β=0.05, expect margin to grow to 0.4–0.6 if the data is clean.

Intervention 3: Reduce learning rate to 1e-7. The oscillating loss at 5e-7 suggests the optimizer is bouncing around the minimum. A 5× reduction stabilizes training. Run for 2 epochs at this lower rate instead of 1 epoch at the original rate.
